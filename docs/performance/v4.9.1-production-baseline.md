- Raw Chrome trace:

- Trace SHA-256:
  `78d7f8402d6ef30301b076c2e5e0b6a1505d930e72a870c5d38517919bd51d61`

## Profile conditions

The controlled profile represents a new mobile user opening the production
Home route:

- viewport: 390 x 844 CSS pixels at device scale factor 2;
- mobile and touch emulation enabled;
- CPU throttled to 4x;
- Chrome `Slow 4G` network profile;
- HTTP cache bypassed for the profiled navigation;
- service worker registrations and Cache Storage cleared;
- FindSpot IndexedDB, local storage and session storage cleared;
- Chrome trace and a buffered `layout-shift` observer started before reload.

The manual trace was stopped after the page had loaded and the startup shifts
had settled. Its post-navigation tail is not part of the measurements below.

## Findings

### 1. Fresh-start Home state causes the dominant layout shift

`Home.tsx` initially has no `permissions` result, so `isFirstRun` is false and
the returning-user Permissions layout is rendered. When the live queries
resolve to an empty new-user database, `isFirstRun` becomes true. The page then
removes returning-user controls and changes the empty card and button spacing,
font size and content.

The instrumented trace attributes 0.0726 of the 0.0796 controlled CLS to this
transition. The Permissions section moves upward by 122 pixels, its empty card
changes size, and the Latest Finds section enters the viewport. A
`transition-all` button then animates layout-affecting properties and contributes
smaller follow-on shifts.

This is consistent with the production field CLS of 0.22 and is the first
bounded Workstream J target.

### 2. The Home entry graph eagerly loads feature-only libraries

The production Home navigation imports and module-preloads PDF, MapLibre and
Turf chunks plus MapLibre CSS even though the initial Home view does not use
those features.

| Initial dependency | Compressed bytes | Decoded bytes |
| --- | ---: | ---: |
| PDF | 190,237 | 639,607 |
| MapLibre JavaScript | 210,082 | 786,568 |
| Turf | 96,877 | 382,120 |
| MapLibre CSS | 9,311 | 65,328 |
| **Total** | **506,507** | **1,873,623** |

These four resources are 64.7% of the measured compressed production resource
body bytes. They are a strong candidate for the next performance slice, but
they should not be combined with the startup-layout change because they cross
route, modal, significant-find and offline-precache boundaries.

### 3. Static cache lifetime is short but is not the first application slice

GitHub Pages served the hashed assets with a 600-second browser cache lifetime.
The trace reported approximately 719.5 kB of cache-policy opportunity. Hosting
cache policy and the PWA precache lifecycle need separate ownership and
verification; they are not part of the proposed v4.9.2 change.

## Proposed v4.9.2 measurement gate

Use this retained production trace as the before profile. Under the same
controlled mobile conditions, the candidate must:

- keep total startup CLS at or below 0.02;
- keep the largest single startup shift at or below 0.01;
- remove the returning-user-to-first-run layout flip;
- record the before and after values in the v4.9.2 release evidence;
- preserve first-run, returning-user, onboarding, offline and backup behaviour.

No implementation, release commit, push, tag or deploy was made while capturing
this baseline.

The verified v4.9.2 candidate comparison is recorded in
[`v4.9.2-candidate-home-layout.md`](v4.9.2-candidate-home-layout.md).
