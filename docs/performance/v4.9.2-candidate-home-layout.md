# v4.9.2 Home startup-layout candidate evidence

Captured on 23 July 2026 after implementing the first Workstream J slice and
before any release commit, push, tag or deployment.

## Scope

The candidate makes Home wait for the two live-query results that decide its
first-run presentation before rendering the variable page layout. It also
limits the Add Permission button to transform transitions so a state change
cannot animate padding, margin, font size, line height or letter spacing.

No data model, persistence, cache, navigation, onboarding, backup or offline
precache behaviour changes.

The candidate trace recorded no layout-shift entries. The retained production
trace and the pre-change browser characterization both attributed the dominant
shift to Home changing from its provisional returning-user Permissions layout
to its first-run layout after live queries resolved.

LCP in the constrained candidate trace was 8,104 ms, compared with 15,000 ms
in the retained production trace. This is not a v4.9.2 release gate because
the candidate was served from a local preview rather than GitHub Pages and the
initial feature-library dependency graph is intentionally unchanged.

## Compatibility and risk

- First-run and returning-user page content and navigation are browser-tested.
- Existing onboarding, backup, restore, FieldGuide, permission, session and
  significant-find browser coverage remains green.
- The Home content area can remain visually empty for the short period while
  its permissions and finds queries resolve. A minimum-height, labelled busy
  region keeps the surrounding shell stable and exposes loading state to
  assistive technology.
- The eager PDF, MapLibre and Turf dependency graph remains the leading
  follow-on performance candidate. It is deliberately outside this release.
- GitHub Pages' short hashed-asset cache lifetime also remains separate hosting
  work.
