# FindSpot architecture hardening programme (v4.6-v4.9)

## Purpose

This document is the durable source of truth for the architecture hardening
programme. It is deliberately release-gated rather than calendar-based. A
workstream may take several patch releases, but the next workstream does not
start until the current workstream's exit criteria have been evidenced and
signed off.

Privacy, offline-first operation, deterministic behaviour and compatibility
with prior v4.x backups remain non-negotiable throughout.

## Release protocol

Each release slice follows the same sequence:

1. Add characterization tests before changing the implementation.
2. Make one cohesive change within the active workstream.
3. Add or tighten automated ratchets where the exit criterion is measurable.
4. Run targeted checks followed by the full checks required for the affected
   surfaces.
5. Update `package.json` and `src/version.ts` for the proposed release.
6. Present a sign-off report before pushing to `main`.
7. After explicit approval, push once and wait for the deploy and semantic tag.
8. Record the released version, commit, tag and evidence here before advancing.

Intermediate commits may live on a programme branch. They do not constitute a
release and must not be pushed to `main` as partial work.

## Sign-off model

- **Release sign-off:** required before every push to `main`. Covers one bounded
  slice, its tests, compatibility assessment, release notes and known risks.
- **Workstream exit sign-off:** required after all exit criteria for a
  workstream are met. The next workstream starts only after this approval.

The approval phrase `Approved - push <version>` authorizes that release only.

## Workstream A - Backup and data architecture

### Exit criteria

- [ ] No backup-related module exceeds approximately 500 lines.
- [ ] Every live Dexie table is centrally classified as `backup` or `excluded`.
- [ ] Every excluded table has a durable reason.
- [ ] A permanent test compares the registry with the live Dexie table list.
- [ ] Backup format versions and compatibility rules are explicit data.
- [ ] A real IndexedDB fixture passes export -> wipe -> restore in CI.
- [ ] The round-trip covers binary and damaged-media behaviour.
- [ ] Historical v4.x backup fixtures remain restorable.
- [ ] Restore remains atomic on validation, media and quota failure paths.
- [ ] Backup reminders escalate using finds changed since the last external
      backup.
- [ ] A restore drill validates and stages without replacing live data.
- [ ] Restore produces a per-table imported/skipped/repaired/damaged report.

## Later workstream exit criteria

### C - Outstanding Questions state machine

- One pure transition function and an explicit state/event table.
- Illegal transitions and terminal supersession invariants are tested.
- Runtime violations record non-fatal `diagLog` warnings.
- Direct lifecycle-status writes outside the transition module fail a permanent
  check.
- Existing question pipeline integration behaviour remains unchanged.

### D - Worker protocol and reusable client

- Request IDs, protocol versions, cancellation, timeout and typed error
  envelopes are shared.
- Terrain and landscape interpretation workers use the client.
- Worker construction is centralized and permanently checked.
- Round-trip, cancellation, timeout and version-mismatch tests pass.

### F - Persistence boundaries

- No engine imports `db`.
- UI reads may use live queries, but writes use intent-named service functions.
- Permanent checks enforce both boundaries without introducing a generic
  repository abstraction.

### I - Diagnostics and integrity checks

- Silent catches are eliminated, apart from the documented diagnostic sink
  exception.
- Shared integrity checks cover orphaned records, dangling permission IDs and
  retired rules.
- Checks run non-fatally after a schema-version change and surface one-line
  status in Settings.

### E - FieldGuide architecture

- `FieldGuide.tsx` is below approximately 500 lines.
- No FieldGuide hook exceeds approximately 300 lines.
- Scan orchestration is headless and consumes the worker client.
- UI modules contain no worker lifecycle or scan orchestration logic.

### J - Performance

- Every optimization begins with a retained realistic profile.
- Every shipped optimization records before/after measurements in its release
  notes.
- Every new cache uses shared TTL, engine-version invalidation and registry
  classification policy.
