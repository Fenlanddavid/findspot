# FindSpot architecture hardening programme (v4.6-v4.9)

## Purpose

This document is the durable source of truth for the architecture hardening
programme. It is deliberately release-gated rather than calendar-based. A
workstream may take several patch releases, but the next workstream does not
start until the current workstream's exit criteria have been evidenced and
signed off.

Privacy, offline-first operation, deterministic behaviour and compatibility
with prior v4.x backups remain non-negotiable throughout.

## Release protocol

Each release slice follows the same sequence:

1. Add characterization tests before changing the implementation.
2. Make one cohesive change within the active workstream.
3. Add or tighten automated ratchets where the exit criterion is measurable.
4. Run targeted checks followed by the full checks required for the affected
   surfaces.
5. Update `package.json` and `src/version.ts` for the proposed release.
6. Present a sign-off report before pushing to `main`.
7. After explicit approval, push once and wait for the deploy and semantic tag.
8. Record the released version, commit, tag and evidence here before advancing.

Intermediate commits may live on a programme branch. They do not constitute a
release and must not be pushed to `main` as partial work.

## Sign-off model

- **Release sign-off:** required before every push to `main`. Covers one bounded
  slice, its tests, compatibility assessment, release notes and known risks.
- **Workstream exit sign-off:** required after all exit criteria for a
  workstream are met. The next workstream starts only after this approval.

The approval phrase `Approved - push <version>` authorizes that release only.
